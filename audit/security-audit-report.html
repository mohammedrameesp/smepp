Security Concepts Explained

Before diving into the findings, let's understand the key security concepts that apply to multi-tenant SaaS applications.

üè¢ Multi-Tenant Architecture

In a multi-tenant system, multiple customers (tenants) share the same application instance but their data must be completely isolated. Each tenant's data should be invisible to other tenants - this is called tenant isolation.

Example: Company A should never see Company B's employees, assets, or payroll data.

üîê Authentication vs Authorization

Authentication = "Who are you?" (login, identity verification)

Authorization = "What can you do?" (permissions, access control)

Example: After logging in (authentication), a MEMBER role can view assets but only ADMIN can delete them (authorization).

üõ°Ô∏è CSRF Protection

Cross-Site Request Forgery (CSRF) is an attack where a malicious website tricks your browser into making requests to another site where you're logged in.

Example: A malicious link could delete your account if you click it while logged in - unless CSRF protection is in place.

üé´ JWT Tokens

JSON Web Tokens (JWT) are signed tokens that contain user information. They're used for stateless authentication - the server doesn't need to look up session data, it just verifies the token signature.

Think of it like a signed letter of authorization that can't be forged.

üîë Impersonation

Impersonation allows super admins to temporarily access a tenant's account for support purposes. This is powerful and requires careful security controls including token revocation and audit logging.

BLOCKER Issues Found & Fixed

These were critical security vulnerabilities that could allow unauthorized access. All have been fixed.

1. Missing Tenant Models in Isolation Filter

FIXED Commit: 43eae38

The Problem

The system uses a TENANT_MODELS array to know which database tables should be filtered by tenant. 20+ models were missing from this list, meaning queries on those tables could return data from ANY tenant!

Before (Vulnerable)

const TENANT_MODELS = [
  'Asset', 'Supplier', 'Subscription',
  // ... only 14 models listed
  // TeamMember, ChatConversation, etc. MISSING!
];

After (Fixed)

const TENANT_MODELS = [
  'Asset', 'Supplier', 'Subscription',
  'TeamMember', 'ChatConversation', 'AIChatUsage',
  'WhatsAppConfig', 'DepreciationRecord',
  // ... all 35 tenant-scoped models now included
];

Why This Matters

Without the tenant filter, a query like prisma.teamMember.findMany() would return ALL employees from ALL companies - a massive data breach!

2. Cron Job Authentication Bypass

FIXED Commit: 43eae38

The Problem

The cron job endpoint had a conditional that allowed unauthenticated access when the CRON_SECRET environment variable was not set.

Before (Vulnerable)

// If secret exists AND doesn't match, reject
if (cronSecret && authHeader !== `Bearer ${cronSecret}`) {
  return new Response('Unauthorized', { status: 401 });
}
// BUG: If cronSecret is undefined, this check is SKIPPED!

After (Fixed)

// Require secret to be set AND match
if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
  return new Response('Unauthorized', { status: 401 });
}
// Now: No secret = always rejected

Lesson Learned

Security checks should fail closed (deny by default). If something is missing or misconfigured, access should be denied, not granted.

3. Unauthenticated Password Reset Endpoint

FIXED Commit: 57781b1

The Problem

The /api/super-admin/set-password endpoint had NO authentication check. Anyone could set any user's password!

Before (Vulnerable)

export async function POST(request: NextRequest) {
  const { userId, password } = await request.json();
  // NO AUTH CHECK - anyone can call this!
  await prisma.user.update({
    where: { id: userId },
    data: { password: await hash(password, 12) }
  });
}

After (Fixed)

export async function POST(request: NextRequest) {
  // Require super admin authentication
  const session = await getServerSession(authOptions);
  if (!session?.user?.isSuperAdmin) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }

  const { userId, password } = await request.json();
  // Now only super admins can reach this code
  await prisma.user.update({...});
}

Lesson Learned

Every API endpoint must have explicit authentication/authorization checks. Never assume a route is "internal only" - always verify the caller's identity and permissions.

Prisma Extension Magic

The system uses a Prisma extension that automatically adds WHERE tenantId = '...' to every query. This means developers don't have to remember to add tenant filters - it happens automatically.

Example: How Queries are Modified

// What you write:

// What actually executes:
const assets = await prisma.asset.findMany({
  where: { tenantId: 'tenant_123' }  // Automatically added!
});

Security Features Implemented

‚úì Brute-Force Protection

After multiple failed login attempts, the account is temporarily locked. This prevents attackers from guessing passwords.

‚úì Password Change Session Invalidation

When a password is changed, all existing sessions are invalidated. This ensures that if an attacker compromises a session, changing the password kicks them out.

‚úì 2FA for Super Admins

Super admins must use two-factor authentication. Even if their password is compromised, attackers can't access the platform without the 2FA code.

‚úì Secure Cookie Configuration

Session cookies are configured with: httpOnly (no JavaScript access), secure (HTTPS only), sameSite: lax (CSRF protection).

Session Security Check

The system periodically checks if a user should still have access:

// Every 2 minutes, check:
// 1. Has the user's password been changed?
// 2. Has the user been deactivated?
// 3. Has the user been deleted?
const SECURITY_CHECK_INTERVAL_MS = 2 * 60 * 1000; // 2 minutes

Why Not Check Every Request?

Checking the database on every request would be expensive and slow. The 2-minute interval balances security (deactivated users are blocked within 2 minutes) with performance (minimal database queries).

Why Strip Token from URL?

If the token stays in the URL, it could appear in:

  • Browser history
  • Server logs
  • Referrer headers sent to external sites

By redirecting immediately and storing in a cookie instead, we prevent token exposure.

Key Lessons Learned

1. Fail Closed, Not Open

When something is missing or misconfigured (like a secret), deny access rather than allow it. The cron auth bypass was caused by failing open when the secret wasn't set.

2. Defense in Depth

Multiple layers of security protect against failures. Even if one layer fails (like the tenant model list), other layers (session validation, role checks) provide protection.

3. Audit Everything Sensitive

Every sensitive operation (impersonation, password changes, admin actions) should be logged. If something goes wrong, you need to know what happened.

4. Assume Nothing

Never assume a route is "internal only" or that code paths are unreachable. Every API endpoint needs explicit auth checks.

5. Regular Audits Matter

Security vulnerabilities can be introduced gradually as features are added. Regular audits catch issues before they become breaches.
